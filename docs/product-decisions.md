# 「第一步」MVP 产品决策说明

## 为了快速实现 MVP 做的简化

- 单用户、本地运行，不做登录注册、多人协作和云同步
- 只覆盖任务录入、任务列表、站内提醒、提醒响应和基础看板
- 只关心“是否愿意开始”，不追踪任务完成状态和完成率
- 不引入复杂 AI、自动排程或重型任务管理系统

这些简化的目的只有一个：先验证“低阻力启动提醒”这个核心闭环是否成立。

## 哪些能力是 mock / 规则版实现

### 站内提醒

- 当前提醒中心是站内模拟提醒
- 没有接入系统通知、微信、短信或原生推送
- 这是一个明确的降级方案，优先保证提醒链路可跑、可演示、可统计

### 提醒策略

- 有 `dueAt` 的任务按“截止前 1 天 / 2 小时 / 20 分钟”生成候选提醒
- 无 `dueAt` 的任务默认在创建后 2 小时给一次轻提醒
- “10 分钟后提醒”直接顺延 10 分钟
- “今天不做”只跳过当天，不会永久取消任务

### 文案生成

- 提醒文案按 `gentle / minimal_action / ddl_push` 三类模板生成
- 不接大模型，不做复杂上下文生成

### 动作提取

- `parsedAction` 使用轻量规则提取
- 优先覆盖“回消息 / 改简历 / 看岗位 JD / 学习资料”等高频场景
- 不做复杂 NLP 和语义解析

## 演示相关的降级方案

- 没有额外做一键“恢复演示数据”的页面按钮
- 演示前统一通过 `npm run db:seed` 恢复样本数据
- 这是刻意保持 KISS 的取舍，避免为了 demo 再加一套管理入口

## 部署相关的降级方案

- 本地开发继续使用 `prisma/dev.db`
- Vercel 公网演示不切换到云数据库，先使用 `prisma/demo.db` 快照
- 运行时把 `prisma/demo.db` 复制到 `/tmp`，供当前实例读写

这样做的原因：

- 改动最小，不需要把整套数据层重写到新的数据库
- 可以先拿到公网演示链接和 Preview Deployment
- 可以保留本地 SQLite 的开发体验

代价也明确存在：

- Vercel 上的数据不是长期持久化的
- 实例冷启动后可能回到 demo 快照状态
- 这适合演示和预览，不适合作为正式生产数据库方案

## 哪些未来可以扩展但当前未做

- 系统级通知和多渠道提醒
- 登录注册、用户体系、跨端同步
- 更细的提醒时间学习和个性化策略
- 更好的任务理解、动作提取和 AI 文案生成
- 完成率追踪、连续行为分析、长期效果归因
- `ReminderEvent` 事件级快照字段，避免历史统计受任务编辑影响

## 为什么当前把“启动接受率”而不是“任务完成率”作为主指标

这个产品的一阶问题不是“做没做完”，而是“能不能开始”。

对于拖延型用户来说，真正的阻力通常发生在启动前：

- 任务太大，不想碰
- 提醒太像命令，天然抗拒
- 当下时机不对，虽然知道该做但不愿开始

所以 MVP 先衡量这件事：

- 用户收到提醒后，是否愿意点击“现在去做”

这就是启动接受率。

如果启动接受率上不去，继续追踪完成率没有意义，因为用户连第一步都没迈出去。先把“开始”做对，再决定是否要扩展到“完成”。

## 当前指标口径

- 有效提醒总次数：`ReminderEvent.eventType = reminder_sent`
- 启动接受次数：`eventType = accept`
- 延后次数：`eventType = delay`
- 拒绝次数：`eventType = reject`
- 启动接受率 / 延后率 / 拒绝率的分母统一使用有效提醒总次数

## 当前已知限制

### 看板聚合限制

- 看板按任务当前的 `reminderStyle` 和 `contextType` 归类历史提醒
- 如果任务后来被编辑，历史提醒会跟着当前字段重新归类
- 正式版更适合在 `ReminderEvent` 中冗余写入提醒风格和任务场景快照

### 事件记录方式

- 提醒中心页面加载时，对当前展示的提醒做 `reminder_sent` 幂等写入
- 幂等键是 `taskId + scheduledFor`
- 响应事件也按同样方式去重，避免重复点击造成重复统计

### 产品边界

- 当前不是完整 GTD 或项目管理工具
- 不支持复杂计划拆解、子任务、协作和完成流转
