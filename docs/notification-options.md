# 通知能力升级选项

这份文档只定义方向，不直接进入实现细节。

## 目标

当前 Web/PWA 版已经能做“设备通知优先”的最小可用接入，但还不等于稳定的设备级提醒。

后续如果继续升级，有两条路线。

## 方案一：Web Push

### 适合什么阶段

适合作为下一步 MVP。

### 需要补的能力

1. Push Subscription 存储
2. 浏览器端订阅与退订
3. VAPID 密钥管理
4. 服务端推送发送逻辑
5. 推送发送记录与失败重试

### 优点

- 继续沿用当前 Web 技术栈
- 成本低于原生 App
- 能比当前“页面打开时轮询”更进一步

### 限制

- 依然受浏览器和系统策略影响
- 不等于原生本地闹钟
- iPhone 上的体验和权限链路仍有限制

## 方案二：原生移动端通知

### 适合什么阶段

适合作为长期正式版本。

### 需要补的能力

1. iOS / Android 原生客户端
2. 本地通知调度
3. APNs / FCM 推送链路
4. 用户设备令牌管理
5. 推送发送与状态追踪

### 优点

- 最接近真正稳定的设备级提醒
- 可以做更完整的铃声、角标、前后台通知体验
- 可以继续延伸到更稳定的“稍后提醒”和多端同步

### 限制

- 开发成本更高
- 需要维护移动端工程
- 发布和权限链路更重

## 建议

### 下一步 MVP

先做 Web Push。

原因：

1. 当前项目本身还是 Web 产品
2. 当前已经有 Notification API / Service Worker 基础
3. 先把“设备通知优先”做完整，比直接开原生工程更划算

### 长期正式版本

原生移动端通知是更终局的方案。

原因：

1. 更稳定
2. 更接近系统级提醒
3. 更适合长期在线产品
