# 通知能力升级选项

这份文档说明当前 Web/PWA 版已经做到什么、还没做到什么，以及后续继续升级的方向。

## 当前 Web/PWA 版已经实现的能力

当前版本已经接入：

1. Notification 权限申请与状态检测
2. Service Worker 注册
3. Push Subscription 获取与数据库保存
4. 服务端 Web Push 发送
5. 到点提醒推送成功后写入 `reminder_sent`
6. 通知点击回到提醒页对应任务

## 当前 Web/PWA 版还做不到什么

1. 不能像原生闹钟那样保证每一次都准点响铃
2. Preview / Local 环境不自带稳定后台调度
3. iPhone 必须先“添加到主屏幕”后才有完整 Web Push 能力
4. 仍然会受浏览器节电、系统通知策略和网络状态影响

## 当前 Web/PWA 的适用设备

### 更适合

1. Android Chrome 一类支持 Push API 的浏览器
2. iPhone / iPad 的主屏幕 PWA
3. 桌面端支持 Service Worker + Push API 的现代浏览器

### 需要注意

1. iPhone 在 Safari 普通标签页里不能按原生 App 的方式稳定收提醒
2. iPhone 需要先“添加到主屏幕”，再从主屏幕启动后开启通知
3. 本地开发更适合验证测试提醒和手动同步，不适合作为稳定自动提醒环境

## 方案一：Web Push

### 当前已经有的基础

1. 浏览器端订阅
2. VAPID 密钥接入
3. 服务端推送发送逻辑
4. 失败订阅的基础失效处理
5. 生产环境 cron 调度入口

### 下一步还适合继续补什么

1. 更稳定的生产调度器
2. 订阅失效重试和清理
3. 推送发送状态看板
4. 更精细的通知节流与去重

### 优点

- 继续沿用当前 Web 技术栈
- 接入成本低于原生 App
- 已经能把提醒从“站内可见”升级到“设备可见”

### 限制

- 依然不等于原生本地闹钟
- 依然受浏览器和系统策略影响
- iPhone 上的权限链路和入口比原生更重

## 方案二：原生移动端通知

### 适合什么阶段

适合作为长期正式版本。

### 需要补的能力

1. iOS / Android 原生客户端
2. 本地通知调度
3. APNs / FCM 推送链路
4. 用户设备令牌管理
5. 推送发送与状态追踪

### 优点

- 最接近真正稳定的设备级提醒
- 可以做更完整的铃声、角标、前后台通知体验
- 更适合长期在线产品

### 限制

- 开发成本更高
- 需要维护移动端工程
- 发布和权限链路更重

## 建议

### 下一步 MVP

继续沿 Web Push 往前补。

原因：

1. 当前项目本身还是 Web 产品
2. 当前已经有 Service Worker、Subscription 和服务端推送基础
3. 先把 Web/PWA 设备提醒做稳，比直接开原生工程更划算

### 长期正式版本

原生移动端通知仍然是更终局的方案。

原因：

1. 更稳定
2. 更接近系统级提醒
3. 更适合长期在线产品

## 什么时候必须转原生

如果你需要下面这些能力，Web/PWA 就不够了：

1. 准点铃声和闹钟级可靠性
2. 更强的前后台本地调度
3. 更深的 iOS / Android 系统通知能力
4. 更稳定的离线提醒和多端同步体验
